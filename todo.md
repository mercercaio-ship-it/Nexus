## Open / deferred backlog

None of the items below are gating; each is opt-in and requires
explicit phase-level approval before work starts.

- **Code signing / signed installer.** Today's Windows installer is
  unsigned (SmartScreen warns on first run). Signing is gated on a
  future signing-certificate decision.
- **Auto-update / `electron-updater`.** Explicit non-goal until the
  signing-certificate decision lands. No automatic update channel
  exists today.
- **GitHub Actions release workflow.** Explicit non-goal until
  signing + tagging policy is decided. Releases are cut manually
  per [`docs/release-readiness.md`](docs/release-readiness.md) §5.
- **CI hygiene** — Dependabot config, CodeQL scanning, a read-only
  build/typecheck workflow. Optional Phase 11-D or later.
- **CODEOWNERS** (optional).
- **macOS / Linux desktop binaries.** Explicit non-goal today —
  the packaged-build target is Windows x64 NSIS only.
- **Admin-console authentication.** Intentional non-goal per §7-D
  in [`docs/roadmap-history.md`](docs/roadmap-history.md). The
  admin surface assumes a trusted local user.
- **Phase 5 deferred extensions** — §5.3-C optional LLM
  summarization, §5.6-B nightly backup scheduling + push,
  §5.6-C destructive restore flow, optional all-agent compaction
  status variant. Opt-in / non-gating.
- **Post-Phase-6 polish backlog** — light/dark theme toggle,
  syntax highlighting in code blocks, hooks split-out, file/image
  attach in composer.
- **v0.1.1 release** to collect the post-v0.1.0 governance / docs
  changes landed on `main` after the publication. Optional —
  follow `docs/release-readiness.md` §5 when the maintainer is
  ready.
- **Documentation hygiene** — the literal user-home path leak at
  `electron/NOTES.md:196` (`cd C:\Users\<you>\OneDrive\...`)
  remains preserved-as-is per the standing don't-touch rule.
  Tracked for a future docs-hygiene turn.

## Rules for maintaining this roadmap

These rules exist because the file became too large to be useful
before Phase 11-C. Keep them in force:

1. **Keep `todo.md` short.** Aim for ≤100–150 lines.
2. **Put detailed evidence in
   [`docs/roadmap-history.md`](docs/roadmap-history.md).** That is
   the archival audit trail.
3. **Do not paste huge raw logs into `todo.md`.** Use the history
   file for log excerpts, PowerShell walkthroughs, and
   sandbox-validation transcripts.
4. **Keep current status accurate.** When a phase or slice closes,
   update the table above with the new status — do not leave stale
   `(review pending)` markers in the current-roadmap section.
5. **Move completed slice details to history.** When closing a
   phase, append the closure footer to
   [`docs/roadmap-history.md`](docs/roadmap-history.md) rather
   than `todo.md`.
6. **Never claim a release is unpublished after it is published.**
   v0.1.0 is published — say so consistently across `todo.md`,
   `CHANGELOG.md`, `docs/release-notes/v0.1.0.md`, and
   `docs/release-notes/README.md`.
7. **Never claim a committed phase is review-pending.** Phase 11-A
   and Phase 11-B are committed and pushed; Phase 11-C is the only
   slice that can still carry a `(review pending)` marker until
   its commit lands.
8. **One small, well-scoped phase / slice per turn.** Each slice
   should land with a closure footer in
   `docs/roadmap-history.md` that records the commit hash, file
   change scope, validation evidence, and privacy/security
   contract enforcement.
9. **Privacy: avoid spreading local paths.** When recording user
   paths in either file, prefer `C:\Users\<you>\…` over the
   literal user-home string unless the literal path is essential
   evidence (in which case keep it inside an archival closure
   footer in the history file, never in the live roadmap).
